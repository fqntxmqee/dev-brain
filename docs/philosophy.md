# 大脑模块底层哲学：多视角辩证 + 文献驱动的深度设计

> 本文档记录了 dev-brain 核心模块设计的底层方法论。这不是一次性规范，而是可复用的思维框架——当你面对任何一个模块时，这套哲学告诉你"怎么想"。

---

## 一、核心信念

好的设计不是"想出来的"，而是**碰撞出来的**。碰撞发生在三个层面：

1. **视角碰撞** — 工程视角看完整性，博弈论视角看激励相容，安全视角看攻击面。单一视角必然有盲区。
2. **信息碰撞** — 大脑训练数据中的"常识" vs 行业最新实践 vs GitHub 上真实项目的设计选择。三者不一致时，信后两个。
3. **层级碰撞** — 全局设计辩论出的共识，下沉到模块级时可能被推翻。全局正确不代表局部正确。

---

## 二、两阶段设计流程

```
阶段 1: 全局设计（做什么 + 为什么不做什么）
  ├─ 多视角独立评审（工程 + 博弈论 + 安全 + ...）
  ├─ 多轮辩证讨论（3 轮 +，直到分歧归零）
  ├─ 共识 → spec 规范
  └─ 未共识 → TODO（明确分歧点和后续讨论时间）

阶段 2: 模块深度设计（怎么做）
  ├─ 2.1 目标锚定: 用户本质目标是什么？
  ├─ 2.2 Grill with Docs: 让大模型带着问题去查行业标准/GitHub/前沿设计
  │      ├─ 第 1 轮: 问题空间探索 — "业界怎么解决这个问题的？"
  │      ├─ 第 2 轮: 方案对比 — "方案 A vs B vs C，在我们的约束下哪个最优？"
  │      └─ 第 3 轮: 详细设计 — "给出可落地的设计，标注关键决策和 tradeoff"
  └─ 2.3 用户确认 → 进入实施
```

### 为什么是 3 轮 Grill with Docs

| 轮次 | 目的 | 典型问题 |
|------|------|---------|
| 第 1 轮 | 打破训练数据的局限 | "这个问题在 2025-2026 年的最佳实践是什么？GitHub 上 star 最高的相关项目怎么做的？" |
| 第 2 轮 | 在约束下做选择 | "方案 A 的优势在我们的场景下是否成立？方案 B 的 tradeoff 我们能否接受？" |
| 第 3 轮 | 收敛到可落地方案 | "给出详细设计。标注每个关键决策的依据是什么（数据/论文/项目）。" |

3 轮不是硬性数字。如果第 2 轮就完全收敛，第 3 轮可以跳过。如果第 3 轮仍有分歧，可以开第 4 轮。但**每轮必须有明确的信息增益**——你不能用相同的输入跑两轮。

## 三、目标锚定原则

在进入任何模块的深度设计前，必须先回答：

### 三层目标递进

```
L1: 用户说想要什么？
    → 例如: "我要 Self-Correction 功能"
    
L2: 用户本质上想达成什么？
    → 例如: "我要任务失败了能自动恢复，减少人工介入"
    
L3: 这个目标的约束条件是什么？
    → 例如: "但不能让 Agent 因为知道会被自动修复而偷懒（道德风险）"
    → 例如: "自动恢复的成功率必须有底线（≥ 40%），否则不如人工介入"
```

如果设计者跳过了 L2 和 L3，直接针对 L1 做设计，就会产生**结构性的设计缺陷**——功能做出来了，但激励机制是反的。CAP-MAR-04 的原始设计就是 L1 思维：用户说要 Self-Correction，就做了一个"失败后自动修复"的工具。L3 思维才发现这会导致 Agent 搭便车。

### 判断标准

当你看到一个设计时，问自己：
- 这个设计如果运行 6 个月，会演化成什么状态？（均衡分析）
- 参与者有没有动机"策略性"地利用这个机制？（激励相容）
- 如果答案让你不舒服，回 L2 重新定义目标。

## 四、多视角辩证原则

### 固定视角矩阵

任何模块设计至少需要以下视角的审视：

| 视角 | 核心问题 | 典型盲区 |
|------|---------|---------|
| **工程视角** | 功能完整吗？设计合理吗？技术先进吗？ | 过度关注"怎么做"而忽略"为什么做" |
| **博弈论视角** | 激励相容吗？参与者会策略性行为吗？长期均衡是什么？ | 假设参与者完全理性，忽略文化/信任因素 |
| **安全视角** | 攻击面在哪？信任边界清晰吗？最小权限原则被遵循了吗？ | 只关注外部攻击而忽略内部滥用 |
| **用户视角** | 用户真的需要这个吗？交互符合直觉吗？失败时用户看到什么？ | 以设计者直觉替代真实用户行为 |

### 视角不是越多越好

视角数量 = 模块的复杂度 × 风险等级。一个只读的 metric 上报模块不需要博弈论审查。一个可以修改自身 prompt 的进化模块需要四个视角全覆盖。

### 视角独立性

每个视角必须**独立形成判断**后再对话。不能让博弈论视角看到工程视角的结论后再写——那样会失去"第二意见"的价值。

## 五、Grill with Docs 的执行原则

### 为什么需要 Grill with Docs

大模型的训练数据有时效边界。一个 2024 年训练的模型可能不知道 2025-2026 年的最佳实践。当你问"怎么设计 X"，它可能给出一个技术上正确但已经过时的方案。

Grill with Docs 强制模型去**查**而非"回忆"：

```
不是: "你怎么设计 X？"
而是: "业界在 2025-2026 年怎么设计 X 的？去 GitHub 找 3 个 star 最高的相关项目，去 Context7 查最新文档，然后对比分析。"
```

### 信息来源优先级

1. **GitHub 真实项目** — 能跑通的代码 > 论文中的方案
2. **官方文档** (Context7 / 原厂 docs) — 最新 API > 训练数据中的旧 API
3. **行业标准/白皮书** — 共识实践 > 单点创新
4. **学术论文** — 有实验数据 > 纯理论
5. **训练数据中的知识** — 仅在前 4 类不可用时作为参考，且需标注"这是训练数据中的知识，可能已过时"

### 每轮的输出格式

每次 Grill with Docs 结束时，必须产出：

```
1. 信息来源清单（哪个 GitHub 项目/哪篇文档/哪个版本）
2. 关键发现（3-5 条，每条注明来源）
3. 方案对比矩阵（至少 2 个候选方案）
4. 推荐方案 + tradeoff 说明
5. 与全局共识的一致性检查（该方案有没有违反之前的共识约束？）
```

## 六、共识管理原则

### 共识的层次

```
全局共识（阶段 1 产出）: 
  → 对 SPEC 的修改（例如: "CAP-EVO-02 必须包含 RuleValidator"）
  → 对模块设计的约束（例如: "Self-Correction 不能只是修复工具，必须有归因"）
  → 模块深度设计时不可推翻，除非你发现新的博弈论证据

模块级共识（阶段 2 产出）:
  → 对实现方案的决策（例如: "RuleValidator Level 1 硬编码, Level 2 YAML 配置"）
  → 可以被新的 Grill with Docs 证据推翻
```

### 共识 ≠ 不再讨论

共识意味着"基于当前信息的最佳判断"。如果后续 Grill with Docs 发现了新的证据（例如 GitHub 上有一个项目做了更优雅的方案），共识可以被推翻。但推翻需要：
1. 新的信息源（不能是同一轮讨论中已经看过的）
2. 明确的推翻理由（为什么新证据优于旧共识）
3. 更新后的设计（不是只否定不建设）

## 七、与 ai-native-os 实践的映射

我们刚完成的 ai-native-os 全局设计，就是这套哲学的首次完整实践：

| 哲学原则 | 实践对应 |
|---------|---------|
| 多视角独立评审 | Claude 工程评审 (`ai-native-os-review.md`) + Codex 博弈论分析 (`game-theory-analysis.md`) |
| 多轮辩证讨论 | 4 轮（Claude R1 → Codex R2 → Claude R3 → Codex R3 → Claude R4 收尾） |
| 共识 → spec | 5 项 P0 共识更新至 `specs/self-evolution/spec.md` 和 `specs/multi-agent-runtime/spec.md` |
| 未共识 → TODO | `review/TODO.md` 记录 3 个开放问题 + 5 个 P2 待讨论项 |
| 目标锚定 L3 | "AI 驱动 AI" → "激励相容的 AI 驱动 AI" → "有熔断和用户反馈兜底的激励相容 AI 驱动 AI" |

### 接下来：模块深度设计

按照这套哲学，每个模块的设计将走"目标锚定 → 3 轮 Grill with Docs → 详细设计"的流程：

```
模块清单（按依赖顺序）:
  ├─ Phase D: Code Observability（代码态观测）
  │     └─ 目标: 为 Phase E 提供数据基础（不是为观测而观测）
  ├─ Phase E 核心: 
  │     ├─ CAP-EVO-02: DiagnosticLLM + RuleValidator
  │     ├─ CAP-EVO-03: EvalRunner + TaskPool
  │     ├─ CAP-EVO-05: CircuitBreaker
  │     └─ CAP-EVO-06: FeedbackCollector
  ├─ Phase E 辅助:
  │     ├─ CAP-EVO-01: InsightEngine
  │     └─ CAP-EVO-04: EvolutionService
  └─ Phase F 关键:
        ├─ CAP-MAR-04: SelfCorrector + AttributionEngine
        ├─ CAP-CTX-01/02/03: 三层记忆 + Recall
        └─ CAP-COM-01: StreamingPusher
```

每个模块将在独立的 `review/module-<name>.md` 中完成深度设计。

---

## 八、一句话总结

**设计不是一次性决策，而是在多视角碰撞中持续收敛的过程。每个模块的起点不是"怎么做"，而是"用户的本质目标是什么，以及如果这个设计运行 6 个月，激励相容的均衡会落在哪里"。**
